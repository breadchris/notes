go dependency injection

March 8, 2024| breadchris
<p>Dependency injection has radically increased my programming rate. When everything is designed as a module, you find yourself constantly reusing things you have already written since it is the path of least resistance.</p> <p>If you try to do this without the help of a framework, you will find yourself writing and rewriting the boilerplate code to connect dependencies together. Most DI frameworks resolve dependency graphs at runtime <a href="https://github.com/uber-go/fx">https://github.com/uber-go/fx</a></p> <p>A dependency graph resolved at runtime is error prone and leads to confusing code. Where does this instance come from? I was excited to find wire, compile-time dependency resolution. <a href="https://github.com/google/wire">https://github.com/google/wire</a></p> <p>The dependency graph is resolved and compiled with a &quot;go generate&quot;. The result is code which clearly communicates what instances are created and where they are going. <a href="https://github.com/justshare-io/justshare/blob/main/pkg/cli/wire_gen.go">https://github.com/justshare-io/justshare/blob/main/pkg/cli/wire_gen.go</a></p> <p>Wire's <a href="https://github.com/google/wire/blob/main/_tutorial/README.md">docs</a> are...complete, but terse. I didn't really understand how to use it until I read June's <a href="https://clavinjune.dev/en/blogs/golang-dependency-injection-using-wire/">blog post</a> on wire.</p>
